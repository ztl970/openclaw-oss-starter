# 异常与 Blocker 处理规则

## 目的

减少 `main` 或 `media` 在可继续执行时过早卡住。

核心原则：

- 能按默认上下文继续的，不要轻易 blocker
- 能先完成局部工作的，先完成局部工作
- blocker 时必须说明“最小缺口”

## 默认上下文优先

当用户没有明确指定其他项目时，默认使用：

- 本地仓库：`/Users/ztl/Documents/New project`
- GitHub 仓库：`ztl970/openclaw-oss-starter`
- 分支：`main`

只要任务与这个项目明显相关，就不应因为“未给 repo”而直接 blocker。

## 可以继续执行的情况

### 1. 缺少外层描述，但默认项目明确

做法：

- 直接按默认项目处理
- 在回报里说明本次基于默认项目上下文执行

### 2. 混合任务中只有公开同步部分不明确

做法：

- 先完成本地仓库工作
- 再把需确认的公开同步部分单独列出

### 3. 知识库页链接缺少，但父目录明确

做法：

- 先按父目录做“存在则更新，不存在则创建”

## 适合 blocker 的情况

### 1. 明显提到另一个仓库

例如：

- 用户给了另一个 GitHub URL
- 用户给了另一个本地路径
- 用户明确说这次不是 `openclaw-oss-starter`

### 2. 有权限风险

例如：

- 无法确认对目标知识库是否有写权限
- 目标 GitHub 仓库不是当前已登录账号可操作范围

### 3. 有高风险误操作可能

例如：

- 同时存在多个候选 repo，且都可能被改动
- 任务会影响错误的公开发布对象

## blocker 时的要求

必须同时给出：

1. 已完成的部分
2. 当前卡住的最小原因
3. 已确认的证据
4. 最小下一步

## 推荐做法

### main

- 优先按默认项目上下文判断
- 优先完成本地可执行部分
- 再决定是否需要派发给 `media`

### media

- 优先按既有知识库父目录、既有发布链路执行
- 能更新则更新，不能更新再明确报权限或回执问题

## 一句话规则

先继续，后阻塞；先明确最小缺口，再请求补充信息。
