# 本地保留内容管理规则

## 目标

规范那些不进入公开链路、但仍然需要在本地保留和管理的内容。

## 适用内容

- 私有脚本
- 个人实验性文件
- 私有运行态内容
- 尚未抽象完成的候选 skill
- 暂不准备发布的工具链文件

## 核心原则

- 本地可保留
- 不自动进入公开仓库
- 不自动进入 ClawHub
- 不自动进入飞书公开项目知识库

## 典型类型

### 1. 私有运行态

- memory
- heartbeat
- 私有会话上下文
- 内部执行记录

处理：

- 只保留本地

### 2. 个人实验脚本

- 尚未稳定的脚本
- 还没判断是否 public-safe 的工具
- 与当前公开项目无直接关系的试验文件

处理：

- 先本地保留
- 与公开提交分开管理

### 3. 未完成抽象的 skill 候选

- 仍然带个人痕迹
- 示例还不够通用
- 结构还不完整

处理：

- 标记为 `D 级`
- 先抽象，再重新判断

## 管理规则

### 不混提

与公开知识库规则或公开项目无关的本地内容，不要顺手混进同一次提交。

### 不默认发布

只要没有通过发布前检查，本地内容就默认不进入：

- GitHub
- ClawHub
- 飞书公开项目知识库

### 单独评估

本地内容若后续想公开，需要重新经过：

- 通用性判断
- public-safe 判断
- 结构检查
- 技能分级与发布决策

## 对当前仓库的适用提醒

如果工作区里出现类似下面这类内容：

- 私有脚本目录
- 试验性工具目录
- 与当前公开项目主线无关的新目录

应优先：

1. 不混进当前公开提交
2. 单独判断是否保留本地
3. 如有潜在公开价值，再进入候选池

## 一句话规则

先本地保留，先不混提；只有完成抽象和检查后，才进入公开链路。
