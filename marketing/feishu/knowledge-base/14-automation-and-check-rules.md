# 自动化与巡检规则

## 目的

这一页用于说明 `main` 如何持续为项目执行自动化检查、复盘和通用指令沉淀。

核心原则：

- 所有自动化都从 `main` 发起
- `main` 负责判断任务类型
- 如涉及公开同步，统一派发给 `media`
- 飞书里的 `小果` 是 `media` 对应的机器人名字

## 当前推荐的自动化

### 1. Main Daily Check

目标：

- 每天检查 GitHub、ClawHub、飞书知识库是否需要同步
- 检查对外简介、知识库页面、发布文案是否存在明显不一致

建议输出：

- 已检查范围
- 是否发现差异
- 是否已派发给 `media`
- 下一步建议

## 2. Main Weekly Review

目标：

- 每周做一次完整复盘
- 检查项目首页、技能说明、ClawHub 发布状态、知识库沉淀情况
- 回看最近一周发布动作和待办

建议输出：

- 本周变化
- 当前一致性情况
- 需要补的页面或发布动作
- 下周优先事项

## 3. Main Command Digest

目标：

- 提炼近期稳定重复出现的通用指令
- 更新 `12_通用指令库`
- 判断哪些内容适合长期公开沉淀

建议输出：

- 本周新增通用指令
- 值得保留的稳定流程
- 需要下沉到知识库的内容

## 调度规则

### 由 `main` 直接处理

- 本地仓库文档更新
- 脚本和规则文件更新
- 新增本地知识库源文件
- 验证和提交判断

### 由 `main` 派发给 `media`

- 飞书知识库页面创建或更新
- GitHub 展示型同步
- ClawHub 说明或发布同步
- 对外发布文案刷新

## 触发方式

### 手动触发

在飞书 `main` 聊天窗口直接发送：

```text
请从 main 执行一次周期性项目同步检查。
要求：
1. 检查 GitHub、ClawHub、飞书知识库是否需要同步。
2. 如需要公开同步动作，统一分配给 media。
3. 返回已执行事项、未执行事项和下一步建议。
```

### 自动触发

适合配置为 recurring automation：

- Daily Check：每天
- Weekly Review：每周
- Command Digest：每周

## 建议节奏

- 日常维护期：
  - Daily Check 保持开启
  - Weekly Review 每周一次
  - Command Digest 每周一次

- 活跃发布期：
  - Daily Check 保持开启
  - Weekly Review 保持
  - 如新 skill 频繁上线，可额外增加按需发布流检查

## 长期目标

- 让 `main` 成为稳定的项目总入口
- 让 `media` 成为稳定的公开同步执行层
- 让 GitHub、ClawHub、飞书知识库三端形成长期一致的运行节奏
