# 标准回报模板

## 目的

统一 `main` 与 `media` 的任务回报格式，方便你在飞书里快速判断：

- 做到了什么
- 谁执行的
- 卡在哪里
- 下一步是什么

## 推荐结构

每次任务完成后，优先按下面结构回报：

### 1. 状态

- `done`
- `in_progress`
- `blocker`

### 2. 动作

一句话说明这次实际做了什么。

示例：

- 已完成本地仓库更新并同步 GitHub 首页
- 已由 media 更新飞书知识库页面
- 已检查 ClawHub 版本状态，当前无需更新

### 3. 证据

给出最关键的证据，优先包含：

- 文件路径
- 提交号
- GitHub 链接
- Feishu 页面链接
- ClawHub skill/version 信息

### 4. 同步范围判断

明确这次的公开同步范围，例如：

- `仅 GitHub`
- `GitHub + 飞书知识库`
- `GitHub + ClawHub`
- `GitHub + ClawHub + 飞书知识库`
- `仅本地保留，不进入公开链路`

### 5. 下一步

只写真正有必要的下一步。

如果没有自然下一步，可以省略。

## 简版模板

```text
状态: done
动作: 已完成本地更新并同步公开侧
同步范围判断: GitHub + 飞书知识库
证据: commit / link / page URL
下一步: 如需继续，可执行 xxx
```

## blocker 模板

```text
状态: blocker
动作: 已完成已知部分检查，但当前缺少必要上下文
同步范围判断: 待判断，当前先不进入公开链路
证据: 已确认的仓库 / 页面 / 当前错误
下一步: 请补充最小必要信息，或按默认项目上下文继续
```

## 适用范围

- GitHub 项目任务
- 飞书知识库同步
- ClawHub 发布检查
- 新 skill 上线流程
- 周度复盘与日常同步检查
- 多项目日常检查

## 多项目场景补充

如果是多项目检查，建议改为：

```text
状态:
动作:

项目 A
- 同步范围判断:
- 动作:
- 证据:
- 下一步:

项目 B
- 同步范围判断:
- 动作:
- 证据:
- 下一步:
```

不要把多个项目混成一个总判断。

## 原则

- 先说结果，再说过程
- 证据尽量短，但要可验证
- blocker 要明确最小缺口，不要泛泛而谈
- 如果默认项目上下文已存在，不要重复要求用户再提供 repo
