# 45_Main 的日更 GitHub 判断规则

## 问题来源

像下面这种说法对 `main` 来说仍然偏模糊：

```text
看看自己一天的工作有没有值得更新到 GitHub 的地方
```

原因是它会遇到两个歧义：

1. “自己一天的工作”是指：
   - 群里的对话与执行记录
   - 还是默认项目 `openclaw-oss-starter` 今天的实际变化

2. “值得更新到 GitHub”是指：
   - 直接修改仓库
   - 还是先做判断并给建议

## 正确目标

让 `main` 去做的是：

- 回看默认项目今天的工作结果
- 判断其中哪些属于可公开、可沉淀、值得更新到 GitHub 的变化
- 再给出建议或执行动作

## 推荐指令

```text
请从 main 回看今天围绕默认项目 openclaw-oss-starter 的工作。
要求：
1. 只看今天与该项目相关的代码、文档、规则、知识库与公开同步工作。
2. 判断其中哪些变化值得更新到 GitHub。
3. 如果只是建议层，请给出建议清单。
4. 如果已经足够明确，请直接给出应更新的文件和同步范围判断。
5. 最后按“状态 / 动作 / 同步范围判断 / 证据 / 下一步”回报。
```

## 更短的群内口令

```text
请从 main 回看今天 openclaw-oss-starter 的工作，判断哪些值得更新到 GitHub。
```

## 期望输出

- 今天识别到的可更新项
- 是否只更新 GitHub
- 是否还要同步飞书知识库
- 是否根本不该进入公开链路

## 一句话规则

不要让 `main` 回看“自己抽象的一天”，而要让它回看“默认项目今天的实际公开价值变化”。
