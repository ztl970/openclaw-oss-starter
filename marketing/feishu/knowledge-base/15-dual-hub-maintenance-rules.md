# 双 Hub 维护规则

## 目标

让 GitHub 与飞书知识库形成长期稳定的双 hub 结构：

- GitHub：公开项目主阵地
- 飞书知识库：内部协同、沉淀、复盘与发布支持阵地

## Hub 定位

### 1. GitHub

用于：

- 公开仓库展示
- 技能与项目结构维护
- release 与 changelog
- 对外链接和可公开引用内容

### 2. 飞书知识库

用于：

- 项目总览沉淀
- 发布文案与申请材料整理
- 账号矩阵与执行规则整理
- 节奏、复盘、通用指令、自动化规则沉淀

## 基本原则

- 所有项目任务先从 `main` 入口进入
- `main` 负责判断任务属于本地代码、公开发布还是知识库沉淀
- 如涉及公开同步，统一派发给 `media`
- 飞书中的 `小果` 是 `media` 对应的机器人名字

## GitHub 需要优先保持一致的内容

- 仓库首页定位
- skill 列表与版本信息
- release / changelog
- 对外简介与说明

## 飞书知识库需要优先保持一致的内容

- 项目总览
- 对外发布文案
- 申请与介绍素材
- 发布记录与复盘
- 通用指令库
- 自动化与巡检规则

## 推荐维护节奏

### 日常

- 检查 GitHub 与飞书知识库是否存在明显不同步
- 如有差异，由 `main -> media` 处理

### 每周

- 做一次整体复盘
- 回看知识库页是否需要补充
- 回看 GitHub 首页、release、skill 页面是否需要更新

### 有新 skill 或新版本时

- 先更新本地仓库
- 再同步 GitHub 展示
- 再同步飞书知识库页面

## 判断规则

### 只改 GitHub

适用于：

- 纯代码修改
- 纯仓库结构调整
- 不影响知识库内容的局部更新

### 只改飞书知识库

适用于：

- 复盘
- 账号矩阵
- 发布节奏
- 执行规则
- 指令沉淀

### GitHub + 飞书都要改

适用于：

- 新 skill 上线
- 新 release
- 首页定位变化
- 对外介绍变化
- 长期工作流规则变化

## 长期目标

- GitHub 始终保持“公开、简洁、可引用”
- 飞书知识库始终保持“完整、结构化、可协同”
- `main` 成为唯一稳定入口
- `media` 成为唯一稳定同步执行层
