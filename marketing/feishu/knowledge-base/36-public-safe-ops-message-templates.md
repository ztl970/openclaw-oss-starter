# Public-safe 运营消息框架

## 目标

把内部运营协同经验压缩成一版可复用、可公开、public-safe 的通用框架。

这份公开页只保留：

- 最小字段骨架
- 最小流程骨架
- 最小回报骨架

不公开项目内的细节 playbook、账号策略、群路由和执行诀窍。

适用场景：

- 内容运营协同
- 任务推进与回报
- 轻量素材生产流程
- 发布确认与复盘框架
- 可供 OpenClaw 或类似系统做结构化提取

## 设计原则

- 不绑定具体账号名
- 不绑定具体群名
- 不绑定具体业务体系
- 只保留可结构化提取的关键字段
- 公开版只给框架，不给完整内部方法论

## 最小消息骨架

```text
【消息类型】
【任务ID】
【主题】
【目标】
【输入资料】
【输出要求】
【当前状态】pending / ready / doing / review / done / archived
【负责人】
【截止时间】
【下一动作】
【回报要求】
```

## 可选扩展字段

- `【主归属】`
- `【协同归属】`
- `【优先级】`
- `【发布渠道】`
- `【归档位置】`
- `【标签】`
- `【结果概览】`
- `【复盘结论】`

## 最小流程骨架

推荐保持：

`提出 -> 判定 -> 结构化下发 -> 执行 -> 回报 -> 复盘 -> 归档`

如果公开场景只需要更轻的版本，至少保留：

`提出 -> 执行 -> 回报`

## 最小回报骨架

```text
状态:
动作:
证据:
下一步:
```

## 公开边界

公开版建议只保留：

- 通用字段名
- 通用状态流转
- 通用回报格式

不建议公开：

- 具体群名和账号矩阵
- 内部路由关系
- 详细执行规则与判断口径
- 可直接复刻内部方法论的长模板

## 使用建议

- 如果只是做公开展示，这一页的框架已经足够
- 如果要落到具体项目，再在本地或私有知识库补细则
- 如果某个项目需要更严格的主归属 / 协同归属规则，应在项目内部规则页单独说明

## 公开定位

这是一份可公开的通用协同框架，不对应任何具体账号、群、家庭或业务体系。
