# 公开同步最小动作集

## 目标

定义每次项目更新后，最少需要完成哪些公开同步动作，避免“已经改了内容，但外部展示没跟上”。

## 适用场景

- skill 新增
- skill 小版本更新
- README / 项目定位更新
- 知识库新增规则页
- 对外简介或营销文案更新

## 最小动作集

### 1. 本地确认

至少确认：

- 本地修改已完成
- 关键文件已更新
- 校验已通过

## 2. GitHub

至少确认其中一项：

- GitHub 首页已同步
- 对应 skill 页面已同步
- 对应 release / changelog 已同步

如果本次改动会影响公开展示，GitHub 是必做项。

## 3. 飞书知识库

至少确认其中一项：

- 项目总览已同步
- 对外发布文案页已同步
- 对应规则页或方法页已同步

如果本次改动影响协同、沉淀或对外表达，飞书知识库是必做项。

## 4. ClawHub

在以下情况下属于最小动作集的一部分：

- skill 内容有明显变化
- skill 版本需要更新
- skill 说明需要同步

如果本次不是 skill 层变化，可以跳过。

## 5. 对外文案

在以下情况下属于最小动作集的一部分：

- 项目定位有变化
- 新 skill 上线
- 新版本发布
- 需要对外同步阶段成果

最小形式可以只是：

- 一条 Feishu 短版
- 一条群内同步摘要

## 推荐最小顺序

```text
本地确认
-> GitHub
-> 飞书知识库
-> （如涉及 skill）ClawHub
-> （如涉及传播）对外文案
```

## 最小回报模板

```text
状态: done
动作: 已完成本次最小公开同步
证据:
- GitHub: ...
- Feishu: ...
- ClawHub: ...
下一步:
- 如需更完整发布，再补文案或复盘
```

## 适合跳过的情况

以下内容通常不要求进入最小公开同步：

- 纯本地试验脚本
- 私有运行态内容
- 未通过发布前检查的候选内容
- 仅本地保留的工具或记录

## 一句话规则

只要外部理解会受影响，就至少同步 GitHub 和飞书知识库；skill 变化再补 ClawHub，传播需要再补文案。
