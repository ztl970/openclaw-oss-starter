# 技能发布审批流

## 目标

让 `main` 在 skill 进入公开链路前，有一套稳定的审批与判断流程。

## 适用范围

- 新 skill 首次公开发布
- 已有 skill 的版本更新
- 原本只在本地或知识库中的 skill 候选转为公开

## 审批流步骤

### 1. 候选进入

来源可以是：

- 新写的 skill
- 从个人场景抽象出的候选 skill
- 已在本地验证过的工作流模板

第一步先进入：

- `24_可发布 Skill 候选池`

### 2. 发布前检查

由 `main` 先做统一检查：

- 是否通用
- 是否 public-safe
- 是否结构完整
- 是否命中“不应公开内容清单”

参考页面：

- `22_Public-safe Skill 规则`
- `23_发布前检查清单`
- `25_不应公开内容清单`

### 3. 技能分级

由 `main` 判断该内容属于：

- `A` 可公开发布
- `B` 仅知识库沉淀
- `C` 仅本地保留
- `D` 先抽象再判断

参考页面：

- `26_技能分级规则`

### 4. 发布决策

由 `main` 输出标准决策：

- 结论
- 分级
- 理由
- 动作
- 下一步

参考页面：

- `27_发布决策模板`

### 5. 执行流转

如果结果为 `A`：

- `main` 负责本地仓库准备
- `main` 负责校验
- 如需公开同步，派发给 `media`
- `media` 负责 GitHub、ClawHub、飞书知识库同步

如果结果为 `B`：

- 不进入 ClawHub
- 通常不进入 GitHub skill 公开链路
- 进入飞书知识库沉淀

如果结果为 `C`：

- 仅本地保留
- 不进入任何公开链路

如果结果为 `D`：

- 回到本地继续抽象
- 补结构后重新申请判断

## 最终回报

每次审批流结束，建议输出：

- 状态
- skill 名称
- 分级结果
- 是否允许公开发布
- 执行动作
- 后续建议

## 一句话规则

先入候选池，再做检查，再做分级，再决定是否公开。
