# LinkedIn Follow-up Post

## Recommended post

After publishing the first overview of `openclaw-oss-starter`, one thing feels
even more important to say clearly:

public-safe is not just a privacy label, it is an operating discipline.

For me, that means separating:

- reusable workflow templates that can be shared
- local runtime state that should stay private
- commercially sensitive work that should not be pushed into the public repo by default

That separation is shaping how I maintain the project now.

The public repository is where I publish reusable OpenClaw skills, deployment
references, and workflow rules that other operators can adapt.

The internal side is where I keep local runtime state, coordination details,
and project-specific material that should not be exposed.

That boundary is becoming just as important as the code itself.

https://github.com/ztl970/openclaw-oss-starter
